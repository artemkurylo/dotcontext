# RFC-001: dotcontext CLI

## Status

Accepted

## Summary

Build a CLI tool called `dotcontext` that scaffolds, validates, generates, and imports project context. The CLI makes the dotcontext standard practical — instead of manually creating directories and files, teams run `dotcontext init` and get started in seconds.

## Motivation

The dotcontext standard defines a `.context/` directory convention, but without tooling, adoption requires manual work:

- Creating the directory structure by hand
- Knowing the correct templates for ADRs, RFCs, runbooks
- Remembering to validate structure in CI
- Manually copying documentation from external tools

A CLI eliminates this friction. It also opens the door to AI-powered features — scanning a codebase to generate initial context, and importing existing documentation from tools like Confluence and Jira via MCP.

## Proposed Commands

### `dotcontext init`

Scaffolds the `.context/` directory structure in the current repository.

```bash
dotcontext init
```

Creates:
```
.context/
  CONTEXT.md          # Starter template with sections to fill in
  plans/
  stories/
  rfcs/
  adrs/
  runbooks/
  guides/
```

Behavior:
- If `.context/` already exists, prints a warning and exits (no overwrite)
- `--force` flag reinitializes, overwriting existing structure
- The generated CONTEXT.md contains template sections (What is this?, Architecture, Tech stack, Team, Key decisions, Status) with placeholder text prompting the user to fill them in

### `dotcontext check`

Validates the `.context/` directory against the dotcontext standard.

```bash
dotcontext check
```

Checks:
- `.context/CONTEXT.md` exists and is non-empty
- CONTEXT.md contains required sections (## headings)
- ADR files in `.context/adrs/` follow naming convention: `NNN-slug.md`
- RFC files in `.context/rfcs/` follow naming convention: `NNN-slug.md`
- All internal markdown links between `.context/` files resolve to existing files

Output: human-readable report listing passes and failures.

Exit codes:
- `0` — all checks pass
- `1` — one or more checks failed

This makes it usable in CI pipelines:
```yaml
- name: Validate dotcontext
  run: dotcontext check
```

### `dotcontext generate` (Milestone 2)

AI-powered context generation. Scans the codebase and produces initial context artifacts.

```bash
dotcontext generate
```

Designed to run inside Claude Code or with a Claude API key. The CLI analyzes:
- File structure and language detection
- Package manifests (package.json, go.mod, Cargo.toml, etc.)
- Configuration files (Dockerfile, docker-compose, CI configs)
- Code patterns (frameworks, database usage, auth patterns)

Generates:
- Populated CONTEXT.md with detected architecture and tech stack
- ADRs inferred from technology choices
- Developer setup guide from build/run configuration

This command requires AI integration and will be specified in a separate RFC.

### `dotcontext import --from <source>` (Milestone 3)

Imports existing documentation from external tools into `.context/`.

```bash
dotcontext import --from confluence
dotcontext import --from notion
dotcontext import --from jira
```

Uses MCP (Model Context Protocol) to connect to external tools, read documents, classify them (ADR, RFC, runbook, guide), convert to markdown, and place them in the correct `.context/` subdirectory.

This command requires MCP connectors and will be specified in a separate RFC.

## Phased Delivery

| Milestone | Commands | Dependencies |
|-----------|----------|-------------|
| 1 | `init`, `check` | None — pure Go, no AI or network |
| 2 | `generate` | Claude API or Claude Code |
| 3 | `import` | MCP connectors |

Milestone 1 ships first as a standalone, useful tool. Milestones 2 and 3 add AI capabilities incrementally.

## Installation

```bash
# From source
go install github.com/artemkurylo/dotcontext/cmd/dotcontext@latest

# Prebuilt binary (GitHub Releases)
curl -sSL https://github.com/artemkurylo/dotcontext/releases/latest/download/dotcontext-$(uname -s)-$(uname -m) -o /usr/local/bin/dotcontext

# Homebrew (future)
brew install dotcontext
```

## Open Questions

1. **Should `dotcontext init` be interactive?** It could prompt for project name, tech stack, etc. and pre-fill CONTEXT.md. Or it could stay non-interactive and just drop a template. Starting non-interactive is simpler and more scriptable.

2. **Should `dotcontext check` support a config file?** Teams might want to customize which sections are required in CONTEXT.md, or disable certain checks. A `.context/.dotcontextrc` could handle this. Defer until there's demand.

3. **Should there be a `dotcontext new adr "Title"` command?** A convenience for creating new ADRs/RFCs with the correct numbering and template. Useful but not essential for milestone 1.
